# Жизненный цикл документа

Краткое описание статусов в таблице `digital_objects.status` и переходов между ними.

## Статусы

| Статус | Смысл |
|--------|--------|
| `UPLOADED` | Файл принят (кратковременно до фиксации хэша). |
| `FROZEN` | Хэш SHA-256 зафиксирован; содержимое считается неизменным («заморозка»). Основной рабочий статус до подачи на согласование. |
| `UNDER_REVIEW` | Документ на многоэтапном согласовании. |
| `APPROVED` | Все внутренние этапы пройдены; ожидается **финальная регистрация администратором** в блокчейне. |
| `REGISTERED_ON_CHAIN` | Запись в смарт-контракте выполнена; есть `blockchain_tx_hash`. |
| `REJECTED` | Отклонён на одном из этапов согласования. |
| `TRANSFERRED` | Права переданы другому владельцу (после on-chain регистрации). |

Устаревшие значения в старых строках могут маппиться через `LEGACY_STATUS_MAP` в коде (например `PENDING_APPROVAL` → `UNDER_REVIEW`).

## Основные переходы

1. **Загрузка** → объект создаётся в `FROZEN` (с событиями `UPLOAD`, `FREEZE` в журнале).
2. **Подача на согласование** (`submit-for-registration`) → `UNDER_REVIEW`; сбрасываются действия по этапам согласования для этого документа.
3. **Согласование** → пока статус `UNDER_REVIEW`; решения фиксируются в `approval_actions`.
4. **Последний этап согласован** → `APPROVED`.
5. **Финальная регистрация** (админ, on-chain) → `REGISTERED_ON_CHAIN`, заполняются `blockchain_tx_hash`, `blockchain_object_id`, `blockchain_registered_at`.
6. **Отклонение на этапе** → `REJECTED` (повторная подача возможна согласно правилам API).
7. **Передача прав** → `TRANSFERRED` (после успешной on-chain регистрации).

## Видимость в общем реестре

В глобальном списке (`/files/global`) не показываются документы только в `FROZEN` / `UPLOADED` — после подачи на согласование они попадают в реестр.

## Роли и операции

| Действие | Кто |
|----------|-----|
| Загрузка, подача на согласование | Владелец (`user`) |
| Подтверждение / отклонение этапа | Роли этапа (`department`, `dean`, `registrar`) или `admin` |
| Финальная запись в блокчейн | `admin` |

## Хранение данных

| Слой | Что хранится |
|------|----------------|
| **PostgreSQL** | Метаданные документа, статус, ссылки на файл (`storage_key`), хэш, владелец, журнал `document_events`, согласования `approval_actions`, события блокчейна `blockchain_events`, логи верификации и т.д. |
| **MinIO / локальное хранилище** | Бинарное содержимое файлов (ключ `storage_key`). При загрузке по умолчанию используется локальное хранилище; конфигурация задаётся `FILE_STORAGE_BACKEND`. |
| **Blockchain** | Идентификатор объекта и хэш контента в смарт-контракте; в БД дублируются `blockchain_tx_hash` и время регистрации. |
