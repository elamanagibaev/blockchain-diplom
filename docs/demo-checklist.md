# Чеклист ручной проверки (демо / защита)

Используйте после миграций БД и (при необходимости) запуска seed: из каталога `backend`:

```bash
python -m app.scripts.seed_demo
```

Учётные записи и имена демо-файлов см. в выводе скрипта и в разделе «Аккаунты» ниже.

---

## Подготовка

- [ ] Backend запущен, БД доступна, переменные окружения заданы.
- [ ] Для сценария **финальной on-chain регистрации**: настроены `WEB3_PROVIDER_URL`, `CONTRACT_ADDRESS` (или файл), ключи владельца контракта. Иначе шаг регистрации вернёт 503 — это ожидаемо для «сухого» стенда.
- [ ] Frontend указывает на тот же API (`VITE_API_BASE_URL`).

---

## 1. Загрузка и фиксация (статус FROZEN)

- [ ] Войти как **владелец** (например `demo.owner@blockproof.local` / `Demo2026!`).
- [ ] **Загрузка документа**: страница загрузки, указать название (обязательное поле), файл PDF/разрешённый тип.
- [ ] В «Мои документы» документ в статусе **FROZEN** (или эквивалент «черновик» в UI).
- [ ] Открыть карточку: отображаются хэш (фрагмент), владелец, отсутствует `blockchain_tx_hash`.

*Быстрый путь:* в seed уже есть `DEMO_01_FROZEN.pdf` — можно показать карточку без новой загрузки.

---

## 2. Подача на согласование

- [ ] Для документа в `FROZEN` нажать отправку на согласование (кнопка в списке или карточке).
- [ ] Статус → **UNDER_REVIEW**; документ появляется в **общем реестре**.

---

## 3. Согласование этап 1 (кафедра)

- [ ] Выйти, войти как **согласующий 1** (`demo.dept@blockproof.local`, роль `department`) или `admin`.
- [ ] В админке / списке ожидающих по этапу `DEPARTMENT_REVIEW` виден документ (например `DEMO_02_ETAP1_kafedra.pdf` из seed — уже на этом этапе).
- [ ] Подтвердить этап **или** отклонить (при отклонении статус `REJECTED` — для позитивного сценария используйте подтверждение).

---

## 4. Согласование этап 2 (деканат / регистратор)

- [ ] Войти как **согласующий 2** (`demo.dean@blockproof.local`, роль `dean`) или `admin`.
- [ ] Документ на этапе `DEAN_REGISTRAR_REVIEW` (seed: `DEMO_03_ETAP2_dekan.pdf`).
- [ ] Подтвердить этап → статус документа **APPROVED**.

---

## 5. Финальная on-chain регистрация

- [ ] Войти как **администратор** (`admin@example.com` / `admin`).
- [ ] Раздел очереди финальной регистрации (`/admin`): документ в **APPROVED** без транзакции (seed: `DEMO_04_APPROVED.pdf`).
- [ ] Выполнить «Зарегистрировать в сети» (или эквивалент в UI).
- [ ] При успехе: статус **REGISTERED_ON_CHAIN**, заполнены хэш транзакции и время регистрации.

---

## 6. Verify по файлу

- [ ] Страница `/verify`, загрузить **тот же файл**, что был зарегистрирован on-chain.
- [ ] Результат **VALID** (если блокчейн и статус согласованы).

---

## 7. Verify по хэшу

- [ ] Скопировать полный SHA-256 из карточки документа или из ответа API.
- [ ] `/verify` — ввод хэша **или** прямой переход `/verify/hash/{sha256}`.
- [ ] **VALID** для зарегистрированного документа; **NOT_FOUND** для случайного хэша.

---

## 8. Публичная verify page

- [ ] Открыть `/verify/doc/{document_id}` без авторизации (ID из карточки или seed-вывод).
- [ ] Отображаются статус, владелец, фрагмент хэша; `is_authentic` / подпись «в реестре» согласованы с `REGISTERED_ON_CHAIN`.

---

## 9. QR

- [ ] В карточке документа (владелец): QR ведёт на `{origin}/verify/doc/{id}`.
- [ ] При необходимости показать переход по ссылке с телефона (та же публичная справка).
