# Дипломный проект: блокчейн-платформа для выпуска и верификации цифровых дипломов

Полноценный full-stack проект для учебного процесса в вузе: кафедра ведёт студентов и оценки, выпускает диплом, документ проходит согласование, после чего его контрольные данные фиксируются в блокчейне. Сам PDF хранится off-chain, а в блокчейн попадают только хэш и служебные метаданные.

## Что делает система

- Автоматизирует выпуск диплома после завершения обучения студента.
- Хранит сам документ вне блокчейна: локально или в MinIO.
- Считает SHA-256 хэш документа и использует его как основу для проверки подлинности.
- Проводит документ через внутреннее согласование.
- После согласования регистрирует документ в локальной EVM-сети.
- Позволяет проверить диплом по файлу, по хэшу и по публичной verify-странице.
- Показывает историю документа и транзакции в локальном explorer.
- Демонстрирует защитное свойство блокчейна: если после регистрации изменить оценку и пересоздать PDF, система обнаружит подмену по несовпадению хэшей.
- Хранит зарегистрированный оригинал документа отдельно от текущей версии, чтобы можно было открыть и сравнить оба PDF.

## Что было доработано в последней версии

В проект добавлен полный демонстрационный сценарий нарушения целостности диплома после on-chain регистрации. Он нужен, чтобы на защите было понятно, зачем системе блокчейн.

Главные изменения:

- добавлена кнопка **“Изменение данных”** для кафедры и отдельная вкладка **“Изменение данных”** для администратора;
- кафедра может имитировать изменение оценки у студента своего вуза;
- администратор может имитировать изменение оценки у студентов всех вузов;
- после изменения оценки система пересоздаёт PDF диплома с новой оценкой;
- текущий PDF заменяется на новую версию;
- исходный PDF сохраняется как **зарегистрированный оригинал**;
- хэш в блокчейне остаётся старым, поэтому проверка фиксирует подмену;
- в smart contract добавляется событие `TRUST_CHAIN_BROKEN`;
- в журнале документа появляется событие **“Блокчейн-проверка выявила изменение данных”**;
- в карточке документа можно открыть текущую версию и зарегистрированный оригинал;
- публичная проверка без авторизации теперь корректно проверяет файл через backend;
- повторные успешные проверки больше не засоряют журнал: хранится одна запись последней проверки.

Это делает демонстрацию более объективной: документ формально остаётся закреплённым за выпускником, но его текущая версия уже не совпадает с той, которая была утверждена и записана в блокчейн.

### 2. `backend/`

FastAPI API, бизнес-логика и работа с БД.

Что находится внутри:

- `app/api/routes/` — REST API;
- `app/services/` — основная логика по документам, согласованию, блокчейну и выпуску диплома;
- `app/models/` — SQLAlchemy-модели;

Локальная сеть Hardhat и smart contract `FileRegistry.sol`.

Контракт хранит:

- `objectId`;
- `fileHash`;
- `owner`;
- `registeredAt`;
- `metadataURI`;
- `currentStatus`;

Небольшой отдельный FastAPI-сервис, который показывает детали локальной blockchain-транзакции по `tx_hash`.

- PostgreSQL — основная БД;
- MinIO — объектное хранилище файлов;
- Docker Compose — поднимает весь стенд одной командой.

## Как работает сценарий диплома

В этом проекте основной поток уже не про “ручную загрузку абстрактного файла”, а именно про выпуск диплома студенту.

### Роли

- `department` — кафедра: создаёт студентов, ведёт оценки, инициирует выпуск.
- `dean` — деканат: подтверждает документ на этапе согласования.
- `admin` — финально регистрирует документ в блокчейне, управляет пользователями и может демонстрационно изменить данные у любого студента.
- `student` — студент/выпускник, которому закрепляется диплом.

### Бизнес-поток

1. Кафедра создаёт студента.
2. Кафедра заносит и обновляет оценки.
3. После завершения обучения вызывается выпуск: создаётся дипломный документ.
4. Документ получает хэш и переходит в состояние `FROZEN`.
5. Документ отправляется на согласование.
6. Деканат подтверждает этап согласования.
7. После этого документ становится готов к финальной blockchain-регистрации.
8. Администратор отправляет контрольные данные документа в smart contract.
9. После on-chain регистрации документ закрепляется за выпускником.
10. Пользователь может проверить диплом:
   - по исходному файлу;
   - по SHA-256 хэшу;
   - по публичной ссылке `/verify/doc/{id}`;
   - по blockchain-транзакции через local explorer.

## Статусы документа

Основные статусы:

- `FROZEN` — документ сформирован, хэш зафиксирован, содержимое считается неизменяемым.
- `UNDER_REVIEW` — документ на согласовании.
- `DEAN_APPROVED` / `APPROVED` — внутренние этапы завершены.
- `REGISTERED_ON_CHAIN` — документ зарегистрирован в блокчейне.
- `ASSIGNED_TO_OWNER` — документ зарегистрирован и закреплён за выпускником.
- `REJECTED` — документ отклонён на одном из этапов.
- `TRANSFERRED` — право/владение передано после on-chain регистрации.

Отдельно от жизненного статуса хранится состояние целостности. Например, документ может оставаться в статусе `ASSIGNED_TO_OWNER`, но при этом иметь нарушенную цепочку доверия, если текущий PDF отличается от зарегистрированной версии.

Подробности: [docs/document-lifecycle.md](docs/document-lifecycle.md), [docs/approval-workflow.md](docs/approval-workflow.md)

## Что хранится где

### В PostgreSQL

- пользователи и роли;
- университеты;
- студенты, прогресс обучения, оценки;
- метаданные документов;
- статусы и этапы согласования;
- журнал действий по документам;
- логи верификации;
- tx hash, object id и другие off-chain сведения о blockchain-операциях.

### В MinIO или локальном storage

- бинарное содержимое файлов;
- шаблоны и результирующие PDF/DOCX-файлы.

### В блокчейне

- хэш документа;
- владелец;
- время регистрации;
- metadata URI;
- история действий по объекту.

Важно: сам файл в блокчейн не записывается.

## Контроль целостности после регистрации диплома

В системе реализован демонстрационный сценарий, который показывает практическую роль блокчейна: он нужен не для хранения PDF, а для доказательства того, что текущий документ совпадает с утверждённой версией.

### Базовая идея

1. Кафедра выпускает диплом студента.
2. PDF диплома формируется автоматически по данным студента и его оценкам.
3. Система считает SHA-256 хэш PDF.
4. После согласования хэш и метаданные документа записываются в smart contract.
5. С этого момента хэш в блокчейне считается эталонным.

Если после регистрации кто-то меняет оценку, система заново формирует PDF. Новый файл становится текущей версией документа, но хэш в блокчейне не меняется. При проверке система сравнивает:

- хэш, закреплённый в блокчейне;
- SHA-256 текущего PDF;
- сведения о последнем событии нарушения цепочки доверия.

Если значения не совпадают, документ считается изменённым после регистрации.

### Зарегистрированный оригинал и текущая версия

При первом изменении данных после on-chain регистрации система сохраняет ссылку на исходный PDF как на зарегистрированный оригинал. Это файл, хэш которого был записан в блокчейн.

После изменения оценки:

- `registered original` остаётся прежним;
- текущий PDF пересоздаётся по обновлённым данным;
- `digital_objects.sha256_hash` начинает соответствовать текущей версии;
- в `document_events.metadata` сохраняются:
  - `registered_hash`;
  - `current_hash`;
  - `registered_original_storage_key`;
  - `registered_original_storage_backend`;
  - `registered_original_file_name`;
  - `registered_original_hash`;
  - `trust_chain_tx_hash`.

Это позволяет открыть оба файла:

- текущую изменённую версию;
- зарегистрированный оригинал, хэш которого закреплён в блокчейне.

Важно: блокчейн хранит не PDF, а хэш оригинального PDF. Поэтому корректная формулировка в интерфейсе и защите: **“зарегистрированный оригинал, хэш которого закреплён в блокчейне”**.

### Где показывается нарушение

Если цепочка доверия нарушена, предупреждение выводится в нескольких местах:

- карточка документа `/files/{id}`;
- публичная страница документа `/verify/doc/{id}`;
- внутренняя вкладка “Верификация”;
- главная публичная проверка без авторизации;
- локальный blockchain explorer для события `TRUST_CHAIN_BROKEN`.

В списках документов основной статус не заменяется на “Данные изменены”. Это сделано специально: жизненный статус документа и результат проверки целостности — разные вещи. Например, диплом остаётся **“Закреплён за выпускником”**, а факт подмены показывается в детальной карточке, верификации и blockchain-событии.

Текст ошибки должен быть на русском языке, например:

```text
Цепочка доверия нарушена.
Текущие данные документа отличаются от версии, зарегистрированной в блокчейне.
```

### Blockchain-событие нарушения

При изменении оценки после регистрации вызывается `appendAction` в smart contract с типом действия:

```text
TRUST_CHAIN_BROKEN
```

В details записываются дисциплина, старая и новая оценка, хэш из блокчейна и текущий хэш. Это показывает, что система не просто сравнила строки в базе, а зафиксировала факт нарушения в blockchain-журнале объекта.

### Две транзакции в карточке документа

В блоке **“Доказательство в блокчейне”** карточка документа показывает исходную транзакцию регистрации:

- `Object ID` — идентификатор документа в smart contract;
- `Tx hash` — транзакция, в которой документ был зарегистрирован;
- кнопка открытия транзакции регистрации в local explorer.

Если после регистрации была обнаружена подмена, в этом же блоке дополнительно отображается отдельное событие:

```text
Событие нарушения целостности
```

У него свой `Tx hash` и отдельная кнопка открытия в обозревателе.

Это важно для объяснения на защите:

- транзакция регистрации доказывает, какой хэш был утверждён изначально;
- событие нарушения показывает, что позже система обнаружила несовпадение текущего PDF с зарегистрированным хэшем;
- прошлое в блокчейне не переписывается, а новая информация добавляется отдельным событием.

### Журнал проверок

Журнал документа не должен заполняться сотнями одинаковых проверок. Поэтому обычная успешная проверка хранится как одна служебная запись последней проверки:

```text
Проверка успешна
```

Если документ проверили успешно ещё раз, эта запись обновляется по времени и metadata, а новая строка не создаётся.

Если позже обнаружена подмена, фиксируется отдельное значимое событие:

```text
Блокчейн-проверка выявила изменение данных
```

Повторные проверки уже изменённого документа не должны создавать новые “Проверка успешна”, потому что документ больше не проходит проверку цепочки доверия.

Логика такая:

- если документ проверен успешно первый раз, создаётся `VERIFY_SUCCESS`;
- если документ снова проверен успешно, эта же запись обновляется по времени;
- если после этого документ был изменён, появляется отдельная запись `VERIFY_FAILED` с `method=demo_data_change`;
- повторные проверки изменённого документа не создают новые успешные события.

Так журнал показывает важные состояния документа, а не каждое нажатие кнопки.

### Как демонстрировать на защите

1. Войти под кафедрой.
2. Выпустить диплом студенту.
3. Дождаться согласования и регистрации в блокчейне.
4. Открыть карточку документа и показать SHA-256 / blockchain transaction.
5. Через вкладку “Изменение данных” у кафедры или администратора изменить оценку.
6. Открыть карточку документа:
   - появится предупреждение о нарушении цепочки доверия;
   - будет доступна текущая версия;
   - будет доступен зарегистрированный оригинал.
   - в блоке “Доказательство в блокчейне” будут видны две операции: регистрация документа и событие нарушения целостности.
7. Открыть публичную проверку и показать, что документ больше не считается подлинным.
8. Открыть blockchain-событие нарушения в local explorer.

Важно для локальной разработки: Hardhat-сеть демонстрационная. Если полностью перезапустить blockchain-контейнер и потерять состояние сети, старые `tx_hash` из БД могут ссылаться на транзакции, которых уже нет в новой локальной сети. Для чистой демонстрации после сброса окружения лучше заново пройти регистрацию диплома.

## Быстрый запуск через Docker

Это основной и рекомендуемый способ запуска.

### 1. Поднять проект

```bash
docker compose up --build
```

Или в фоне:

```bash
make up
```

### 2. Открыть сервисы

- Frontend: `http://localhost:5173`
- Backend API docs: `http://localhost:8000/api/docs`
- Local Explorer: `http://localhost:4000`
- MinIO Console: `http://localhost:9001`
- Hardhat RPC: `http://localhost:8545`

### 3. Что произойдёт автоматически

При старте:

- поднимется PostgreSQL;
- поднимется MinIO;
- запустится Hardhat node;
- smart contract автоматически задеплоится;
- адрес контракта и ABI будут записаны в shared volume;
- backend дождётся БД;
- backend выполнит `alembic upgrade head`;
- backend попробует создать `admin@example.com / admin`, если такого пользователя ещё нет.

## Локальный запуск без Docker

Можно, но сложнее. Понадобятся:

- Python 3.11;
- Node.js 20+;
- PostgreSQL;
- при необходимости локальный Hardhat;
- при необходимости MinIO.

### Backend

```powershell
cd backend
python -m venv .venv
.\.venv\Scripts\python -m pip install --upgrade pip
.\.venv\Scripts\pip install -r requirements.txt
copy ..\.env.example .env
alembic upgrade head
uvicorn app.main:app --reload --port 8000
```

### Frontend

```bash
cd frontend
npm install
npm run dev
```

### Blockchain

```bash
cd blockchain
npm install
npx hardhat node
```

В отдельном окне:

```bash
cd blockchain
```

## Коротко: что показывать на защите

Самый понятный demo-flow:

1. Войти как кафедра.
2. Показать студентов и оценки.
3. Выпустить диплом студенту.
4. Показать документ в статусе `FROZEN` или `UNDER_REVIEW`.
5. Войти как деканат и подтвердить этап.
6. Войти как админ и зарегистрировать документ в блокчейне.
7. Открыть карточку документа, tx hash и local explorer.
8. Открыть публичную страницу проверки и показать, что документ подтверждается.
9. Вернуться под кафедру или администратора и изменить оценку через “Изменение данных”.
10. Показать, что текущий PDF изменился, но blockchain-хэш остался от исходной версии.
11. Открыть карточку документа:
    - текущую версию;
    - зарегистрированный оригинал;
    - транзакцию регистрации документа;
    - blockchain-событие `TRUST_CHAIN_BROKEN`.
12. Проверить изменённый документ через `/verify` или главную страницу и показать ошибку цепочки доверия.

Главная мысль для защиты: блокчейн не запрещает кому-то изменить данные в БД или пересоздать PDF, но позволяет доказать, что текущий документ уже не совпадает с утверждённой версией.
